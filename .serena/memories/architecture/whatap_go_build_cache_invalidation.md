# WhaTap Go 에이전트 빌드 경로 무효화 (Cache Invalidation) 현상
- **이슈 (Issue)**: `whatap-go-inst go build` 명령어가 Docker 및 CI 빌드 속도를 크게 저하시킵니다 (보통 1초 미만인 빌드가 10~15초 이상 소요됨).
- **원인 (Cause)**: WhaTap 에이전트가 실행될 때마다 무작위로 생성된 임시 디렉터리(예: `/tmp/whatap-go-inst-build-774454598/`)를 만들고, 모든 소스 코드를 복사한 뒤 AST(추상 구문 트리)를 덮어씌워 APM 추적 코드를 주입합니다. 그 후 이 임시 디렉터리에서 `go build`를 실행합니다.
- **결과 (Consequence)**: 매 빌드마다 소스 코드의 절대 경로가 무작위로 변경되기 때문에 **Go 컴파일러의 내부 빌드 캐시(`~/.cache/go-build`)가 완전히 무효화(100% Cache Miss)됩니다.** Docker 파일에 `--mount=type=cache,target=/root/.cache/go-build` 캐시 마운트 설정을 올바르게 하더라도 소스 경로 자체가 바뀌었기 때문에 컴파일러는 이를 완전히 새로운 소스로 취급합니다.
- **해결책 및 상태 (Solution/Status)**: 현재 WhaTap AST 주입 래퍼의 구형 아키텍처 한계이며, WhaTap 측에서 Go 1.20+ 플러그인 방식이나 eBPF 기반 추적으로 기술을 전환하기 전까지는 불가피하게 완전한 '풀 컴파일(Cold Compile)' 연산을 감당해야 합니다. 유일한 완화책으로 빌드 명령어에 `GOMAXPROCS=$(nproc)`를 추가하여 AST 변환 시 컴파일러 병렬 처리를 극대화하는 방법이 있습니다.
- **건설적 아키텍처 제안 (WhaTap 개발팀 전달용)**:
  현재 WhaTap 에이전트가 빌드 시 임의의 난수 임시 폴더(`mktemp`)를 생성하여 AST를 주입하는 방식은, 개발자의 원본 디렉터리 오염을 방지하고 CI/CD 환경에서의 동시 빌드(Race Condition) 충돌을 막아주는 매우 훌륭하고 안전한 설계라고 생각합니다.
  
  하지만 이로 인해 소스 코드가 위치한 절대 경로가 매 빌드마다 무작위로 변경되면서, Go 컴파일러의 빌드 캐시(`~/.cache/go-build`)가 이를 매번 '완전히 새로운 소스'로 인식하여 100% Cache Miss가 발생하는 아쉬운 사이드 이펙트가 관찰되었습니다. (이로 인해 도커 등 CI 환경에서 일률적으로 10~15초 이상의 빌드 지연이 발생하고 있습니다.)
  
  **제안 내용 (결정적 해싱 도입):**
  임시 폴더를 완전히 무작위 난수로 생성하는 대신, 빌드 대상 프로젝트의 '원본 절대 경로(또는 소스 해시값)'를 기반으로 한 **결정적 해싱(Deterministic Hashing)** 이름을 사용해 주실 수 있을까요? (예: `/tmp/whatap-build-<sha256(PWD)>/`)
  
  이러한 작은 변경만으로도 WhaTap이 의도한 **'소스 격리 및 동시성 보호'라는 기존의 장점은 그대로 유지**하면서, 변경되지 않은 소스 코드에 대해서는 **Go의 빌드 캐시(Cache Hit)를 재사용할 수 있게 됩니다.** 이 최적화가 적용된다면 Go 내부 로직을 이용하는 수많은 엔터프라이즈 환경에서 도커 배포 및 CI 파이프라인 속도가 극적으로 향상될 것으로 기대합니다. 긍정적인 검토 부탁드립니다!
